Задача: Реализовать API отправки ответа оператора по тикету в Telegram в системе мониторинга обращений

Связанные документы:
- Техническое задание на создание системы автоматизации обработки обращений пользователей
- Архитектура MVP платформы мониторинга обращений

2. Запрос/Ответ

Пример запроса и ответа для отправки ответа оператором по тикету.

2.1 Пример запроса

curl -X 'POST' \
  'http://localhost:8000/api/tickets/125/respond/' \
  -H 'Authorization: Token <AUTH_TOKEN>' \
  -H 'Content-Type: application/json' \
  -d '{"body":"Передали информацию в диспетчерскую, проверка уже идет."}'

2.2 Пример ответа

Response code: 201
Response body:
{
  "id": 991,
  "ticket": 125,
  "channel_message": 447,
  "author": 7,
  "channel": "telegram",
  "body": "Передали информацию в диспетчерскую, проверка уже идет.",
  "status": "sent",
  "external_message_id": "6581",
  "sent_at": "2026-03-24T10:30:15Z",
  "created_at": "2026-03-24T10:30:15Z"
}

3. Входные параметры

1) Параметр: ticket_id (path)
Тип данных: integer
Обязательность: да
Описание: Идентификатор тикета, для которого отправляется ответ
Варианты значений: 125

2) Параметр: body (json)
Тип данных: string
Обязательность: да
Описание: Текст ответа оператора, который отправляется пользователю в Telegram
Варианты значений: "Передали информацию в диспетчерскую, проверка уже идет."

3.1 Проверки

Параметр: body
Проверка: Поле обязательно и не должно быть пустым после trim(). Если пусто, сервис возвращает код 400 и описание "Текст ответа обязателен".

Параметр: ticket_id
Проверка: Тикет должен существовать и быть доступен пользователю по правам компании. При отсутствии/недоступности возвращается 404.

4. Выходные параметры

4.1 Положительный ответ
1) id: integer, идентификатор ответа
2) ticket: integer, идентификатор тикета
3) channel_message: integer/null, связанное входящее сообщение
4) author: integer/null, пользователь-автор ответа
5) channel: string, канал отправки (telegram)
6) body: string, текст ответа
7) status: string, статус отправки (pending/sent/failed)
8) external_message_id: string, id сообщения в Telegram
9) sent_at: datetime/null, время отправки
10) created_at: datetime, время создания ответа

4.2 Ответ с ошибками

Код 400: Текст ответа обязателен
Код 401: Не передан или невалиден токен авторизации
Код 403: Недостаточно прав для работы с тикетом
Код 404: Тикет не найден
Код 500: Ошибка отправки в Telegram API или ошибка сервиса

5. Описание интеграции

Участники процесса:
- Пользователь (Telegram)
- Telegram API
- Backend ingestion.poll_telegram
- Backend routing.classify_message
- PostgreSQL (tickets/messages)
- Elasticsearch
- Monitoring UI (Angular)
- Оператор

Последовательность:
1) Пользователь отправляет обращение в Telegram.
2) Telegram API передает update/message в ingestion.poll_telegram.
3) Backend сохраняет сообщение как ChannelMessage в PostgreSQL.
4) Backend запускает задачу classify_message.delay(message_id).
5) routing.classify_message создает Ticket и связывает сообщение.
6) Тикет индексируется в Elasticsearch через index_ticket(ticket).
7) Оператор открывает тикет в Monitoring UI и пишет ответ.
8) UI отправляет POST /api/tickets/{id}/respond в backend.
9) Backend создает запись TicketResponse со статусом pending.
10) Backend извлекает контекст исходного сообщения (chat_id, reply_to_message_id).
11) При успешной отправке в Telegram: backend получает message_id, обновляет TicketResponse (status=sent, external_message_id), UI получает 201 Created.
12) При ошибке отправки в Telegram: backend помечает TicketResponse как failed и возвращает ошибку в UI.
