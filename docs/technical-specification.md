## 2. Цели и назначение создания системы

### 2.1. Назначение

Система предназначена для сбора обращений пользователей из внешних каналов (мессенджеры, соцсети), их автоматической классификации, маршрутизации и отображения в едином окне мониторинга с возможностью ответа операторами и контроля сроков обработки (SLA).

В рамках проекта автоматизируются: приём обращений из каналов (на MVP — Telegram), классификация по категориям и приоритетам, создание и ведение тикетов, индексация для поиска, расчёт аналитических показателей (MTTA, MTTR), формирование ответов и публикация их обратно в канал.

### 2.2. Цели создания

Система создаётся с целью:

- автоматизации приёма, классификации и маршрутизации обращений пользователей по категориям, приоритетам и группам обработки;
- обеспечения единого окна мониторинга обращений с фильтрацией по каналам, компаниям, периодам и статусам;
- сокращения времени первичной обработки (MTTA) и времени решения (MTTR) за счёт автоматической классификации и контроля SLA;
- упрощения формирования аналитики и отчётности по обращениям для принятия управленческих решений.

В результате создания платформы «Обращения-Мониторинг» должны быть улучшены следующие показатели:

- время от поступления обращения до первой реакции (сокращение за счёт автоматической классификации и уведомлений);
- время поиска и фильтрации обращений (ускорение за счёт единого интерфейса и полнотекстового поиска);
- прозрачность соблюдения SLA (контроль дедлайнов и предупреждения о нарушениях).

## 3. Характеристика объекта автоматизации

Объектом автоматизации является процесс обработки обращений пользователей (пассажиров, клиентов) в части приёма, учёта, классификации и ответа.

Выделены следующие процессы, подлежащие автоматизации:

- сбор и выгрузка обращений из внешних каналов (Telegram: каналы, обсуждения, личные сообщения боту; в перспективе — VK, email и др.);
- автоматическая классификация обращений по категориям, приоритетам и тональности (на базе ключевых слов и ML-модели);
- создание тикетов, назначение групп и контроль сроков обработки (SLA);
- индексация обращений для быстрого поиска и аналитики;
- ответы операторов и публикация ответов обратно в канал (на MVP — в Telegram);
- формирование аналитики (MTTA, MTTR, распределение по категориям и тональности) для отчётности.

## 4. Требования к системе

### 4.1. Требования к структуре и функционированию системы

В состав системы входят:

- **Backend (Django):** приложения `tickets` (тикеты, назначения, ответы), `ingestion` (коннекторы каналов, задачи сбора), `routing` (классификация, SLA), `analytics` (метрики, индексация), `companies` (организации, пользователи, боты). Результирующие сущности: тикет, сообщение канала, ответ — формируются на основе данных из каналов и правил классификации.
- **Очереди и фоновые задачи:** Celery (брокер Redis) для опроса каналов, классификации сообщений и контроля SLA.
- **Хранилища:** PostgreSQL — транзакционные данные; Elasticsearch — полнотекстовый поиск по обращениям; Redis — брокер Celery и кэш.
- **Frontend:** веб-приложение (Angular, Material, ngx-charts) для мониторинга обращений, фильтрации, ответов и просмотра аналитики.

**Взаимодействие фронтенда и бэкенда:** бэкенд на Django предоставляет **REST API** (Django REST Framework) по префиксу `/api/`. Все запросы чтения и записи со стороны Angular выполняются по протоколу HTTP к этому API: тикеты (`/api/tickets/`), сообщения, назначения, ответы, аналитика (`/api/analytics/metrics/`), поиск (`/api/search`), компании и авторизация. Аутентификация — по токену (Token Authentication). Документация API доступна по адресу `/api/docs/` (OpenAPI/Swagger). Фронтенд обращается к API через `HttpClient` с базовым URL из конфигурации окружения.

Система должна обеспечивать: приём обращений из настроенных каналов, автоматическую классификацию и создание тикетов, интерактивную работу операторов с тикетами и ответами, контроль SLA и выдачу аналитических метрик через API и интерфейс.

### 4.2. Требования к интерфейсам

Платформа «Обращения-Мониторинг» представляет собой веб-приложение с интерфейсом, адаптированным для работы на десктопах и планшетах.

Основные элементы интерфейса:

- **Главный экран:** единое окно мониторинга обращений с таблицей тикетов, фильтрами (компания, канал, период, статус, категория, приоритет) и панелью аналитики (MTTA/MTTR, доля обращений по категориям, распределение по тональности). Карточки KPI и графики (линейные, гистограммы) для ключевых метрик.
- **Фильтры и срезы:** многоуровневые селекторы (период, канал, компания, категория, приоритет, статус тикета).
- **Визуализация данных:** таблицы с детализацией по обращениям; графики динамики и распределений; карточки с ключевыми числами (остатки открытых тикетов, среднее время ответа).
- **Интерактивные элементы:** открытие тикета и ответ пользователю из интерфейса; экспорт отчётов (при реализации); детализация при выборе строки/графика.
- **Роли и доступ:** интерфейс различает роли оператора, администратора компании и супер-администратора с соответствующей фильтрацией данных по компаниям.

Интерфейс должен обеспечивать интуитивную работу без предварительного обучения; время на типовую операцию (поиск обращений по срезу и первичный ответ) — не более 2 минут.

### 4.3. Требования к минимальному количеству сотрудников

Минимальное количество сотрудников, требуемое для корректного отображения аналитики и проверки работы системы, должно составлять не менее 2 человек (условно, для тестирования сценариев оператора и администратора). Фактически система ориентирована на работу нескольких операторов и администраторов компаний; данное требование носит справочный характер.
