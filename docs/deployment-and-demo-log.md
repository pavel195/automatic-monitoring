# Журнал развёртывания и демонстрационной настройки

Документ фиксирует практические действия по развёртыванию стенда, настройке
интеграций и проверке демонстрационных сценариев. Секреты, токены и пароли в
журнал не записываются.

## Контур стенда

- Дата работ: `2026-05-22`.
- Сервер: `chipkineev.ru`.
- Серверный каталог проекта: `/home/oem/automatic-monitoring`.

- Elasticsearch на этом стенде не поднимается.

## Развёртывание

1. Для серверного запуска выделен deploy-compose без Elasticsearch.
2. На сервере создан `.env` с production-настройками Django, PostgreSQL,
   Redis и отключённым Ollama.
3. Запуск проверен для сервисов:
   - `backend`;
   - `frontend`;
   - `celery_worker`;
   - `celery_beat`;
   - `postgres`;
   - `redis`.
4. Так как сервер не смог стабильно получить нужные образы из внешнего
   registry, для первичного запуска использовался перенос локально собранных
   `linux/amd64` образов.
5. Серверный checkout переведён на SSH remote Git, чтобы дальнейший rollout
   проходил через локальные изменения, push в `main` и `git pull` на сервере.

## Интеграции

### Telegram

- Для компании создана запись Telegram-бота.
- Токен был проверен отдельно перед привязкой.
- Серверный polling видит активного Telegram-бота.
- На момент проверки сервер не может установить исходящее соединение с
  `api.telegram.org`, поэтому Telegram polling получает сетевую ошибку и не
  забирает реальные сообщения.
- В логах сетевой ошибки Telegram был обнаружен риск раскрытия токена через URL
  запроса.
- Кодовая правка маскирует токен в новых exception-логах connector.
- Так как старые логи уже могли содержать токен, токен бота нужно
  перевыпустить отдельно.

### VK

- Для компании привязано VK-сообщество `ООО Поездка лакшери`.
- Токен сообщества проверен через VK API.
- Идентификатор сообщества на момент проверки: `238973554`.
- Первый polling VK вернул ошибку выключенного Long Poll API.
- После включения Long Poll статус интеграции возвращён в `active`.
- Повторный polling подтвердил рабочий цикл: worker видит активную VK
  интеграцию и получает ответ VK без ошибки.

## Текущие ограничения и следующие проверки

1. Для VK должны быть включены события Long Poll, включая входящие сообщения.
   Если `message_new` выключен, polling работает, но обращения не поступают.
2. После включения событий VK нужно отправить тестовое сообщение и проверить
   цепочку `ChannelMessage -> Ticket -> UI`.
3. Для Telegram требуется восстановить сетевой доступ сервера к Telegram API.
4. Для отчётной демонстрации полезно иметь заполненные обращения из VK и
   Telegram с несколькими сообщениями в одном обращении.

## Улучшение демонстрационного сценария

- В маршрутизации обращений добавлен сценарий, где новые сообщения одного
  автора в одном канале и компании попадают в уже открытое обращение.

- После rollout на серверный стенд добавлены четыре демонстрационных обращения
  и десять сообщений из VK и Telegram без записи токенов и паролей в
  репозиторий.
- Интерфейс списка обращений переведён с правой панели деталей на модальное
  окно обращения; стартовый блок выбора обращения убран.
- Из левого меню убрана группа интеграций, чтобы рабочее меню осталось
  коротким: главная, обращения, аналитика.

## CI/CD

- Workflow `.github/workflows/ci-cd.yml` запускается для pull request, push в
  `main` и вручную.
- CI проверяет backend через Ruff, frontend через ESLint, production-сборку
  Angular и backend pytest в compose-окружении.
- Deploy job зависит от успешного lint и backend test job.
- Deploy собирает `linux/amd64` образы на runner, передаёт их на сервер по SSH,
  делает fast-forward `git pull` в `/home/oem/automatic-monitoring` и запускает
  `docker-compose.deploy.yml` без серверного build.
- Такой rollout не зависит от исходящего доступа сервера к Docker Hub, который
  на стенде уже блокировал сборку базовых образов.

Для включения автоматического server deploy в GitHub нужны:

1. repository variable `ENABLE_SERVER_DEPLOY=true`;
2. secret `DEPLOY_SSH_PRIVATE_KEY` с приватным ключом пользователя deploy;
3. secret `DEPLOY_KNOWN_HOSTS` со строкой host key сервера.

Публичный ключ deploy должен быть добавлен на сервер для пользователя `oem`.

## Договорённый workflow изменений

Кодовые изменения для стенда выполняются локально:

1. правка и проверка в рабочем дереве;
2. commit и push в `main`;
3. `git pull` и rollout на сервере;
4. sanity-check по UI, API и логам после обновления.
